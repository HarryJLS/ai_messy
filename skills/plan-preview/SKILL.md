# Plan Preview

以资深架构师视角，通过代码探索 + 多轮问答，输出可直接喂给 `/plan-init` 的技术方案文档 `task.md`。

## 定位

`/plan-preview` 是 `/plan-init` 的**前置环节**：

| 阶段 | 工具 | 产出 |
|------|------|------|
| 方案预研 | `/plan-preview` | `task.md`（技术方案文档） |
| 任务初始化 | `/plan-init` | `features.json` + `dev-YYYY-MM-DD.log` |
| 任务执行 | `/plan-next` | 代码实现 |

## 设计原则

- **架构师视角**：不是简单记录需求，而是深入代码库理解现状后给出专业方案
- **需求深挖**：通过系统化提问确认需求细节，不在模糊需求上构建方案
- **协作式决策**：每个关键决策点都与用户确认，不擅自拍板
- **代码驱动**：方案基于实际代码探索，而非凭空设计
- **可执行性**：task.md 格式与 `/plan-init` 完全兼容可直接使用，且详细程度要达到——一个不了解业务背景的技术人员，仅凭此文档就能精准知道改哪里、怎么改、怎么验证
- **场景适配**：针对五大场景采用不同的分析策略
- **渐进式输出**：每个阶段完成后将成果写入计划文件，用户直接看文档比看对话更直观，也便于针对性修改

## 五大场景

| 场景 | 核心关注点 | 探索重点 |
|------|-----------|---------|
| 新功能开发 | 在哪加？怎么加？影响什么？ | 相关模块结构、现有抽象、扩展点 |
| 性能/逻辑优化 | 瓶颈在哪？怎么改？改了会不会引入新问题？ | 热点代码路径、调用链、数据流 |
| 线上问题修复 | 根因是什么？修复方案？如何防止复发？ | 问题代码、上下游依赖、异常处理链 |
| 项目结构优化 | 当前结构问题？目标结构？迁移路径？ | 模块划分、依赖关系、分层架构 |
| 中间件/工具创建 | 解决什么问题？边界在哪？API 怎么设计？ | 现有工具类、使用场景、接口契约 |

## 协议

### 步骤 0: 进入 Plan Mode

skill 触发后，**立即调用 `EnterPlanMode` 工具**进入计划模式。

全程在 plan mode 中完成步骤 1-4 的探索、分析和方案构建。最终在步骤 5 调用 `ExitPlanMode` 提交方案供用户审批。

**Plan mode 的价值**：
- 方案预研阶段只读不写，plan mode 天然适合
- 用户可以在方案确定前反复调整，零成本
- 所有文件写入操作在 plan mode 退出并获用户审批后执行

**渐进式文档输出**：
- 每个步骤完成后，将该步骤的结构化产出写入计划文件（plan file）
- 这里的"计划文件"指 Plan Mode 自带的 plan file（草稿区），用户在对话界面可实时查看
- 步骤 6 才会将计划文件的内容正式写入项目根目录的 task.md
- 计划文件使用 task.md 的目标格式组织内容，确保最终可直接复制为 task.md
- 用户可以针对文档内容提修改建议，修改满意后再继续下一步

### 步骤 1: 需求收集、场景识别与需求深挖

1. **获取需求描述**：

   如果用户在触发时已提供了需求描述或文档路径，直接进入分析。
   否则，询问用户：

   ```
   请描述你要做的事情：
   - 可以直接说需求
   - 可以提供需求文档路径（我会读取并分析）
   - 可以提供飞书/语雀文档链接
   ```

   ⛔ **门控**: 等待用户提供需求信息

2. **场景识别**：

   根据需求自动识别所属场景，并告知用户：

   ```
   🎯 场景识别：[新功能开发 / 性能优化 / 线上问题修复 / 项目结构优化 / 中间件创建]
   理由：[简述为什么归为该场景]
   ```

   如果场景判断不确定，列出候选场景让用户选择。

3. **理解确认**：

   ```
   📋 我的理解：
   - 要做什么：[复述核心目标]
   - 核心诉求：[用户最关心的点]
   - 预期产出：[最终交付物]

   ❓ 不确定的地方：[列出理解模糊的点，主动追问]

   请确认是否正确，或指出理解偏差。
   ```

   ⛔ **门控**：用户确认理解正确后，才能继续

4. **需求深挖**（通过提问确认细节）：

   根据识别的场景，从问题库中挑选关键问题，与用户确认细节。

   **提问原则**：
   - 只问对方案有实质影响的问题（能改变技术选型或实现路径的）
   - 跳过用户在需求描述中已明确回答的问题
   - 每轮最多 3-5 个问题，避免问答疲劳
   - 如果一轮问完仍有不确定项，可追加一轮

   **场景化问题库**：

   | 场景 | 必问（3-4 个，直接影响方案方向） | 按需（2-3 个，视情况补充） |
   |------|--------------------------------|--------------------------|
   | 新功能开发 | ① 目标用户角色和核心使用场景是什么？② 数据从哪来、到哪去、关键字段有哪些？③ 项目中有没有类似功能可参考？④ 与现有模块的交互边界在哪？ | ⑤ 并发/性能有特殊要求吗？⑥ 需要向后兼容已有接口吗？⑦ 错误/异常场景怎么处理？ |
   | 性能/逻辑优化 | ① 当前性能数据是什么（QPS/耗时/内存）？瓶颈表现在哪？② 目标性能指标是什么？③ 可接受的改动范围有多大？④ 是否需要降级/兜底方案？ | ⑤ 有没有历史优化尝试（成功或失败的）？⑥ 读写比例大概是多少？⑦ 缓存策略有偏好吗？ |
   | 线上问题修复 | ① 问题的复现条件是什么？② 影响范围和紧急程度？③ 需要先出临时方案还是直接根治？④ 回归测试怎么做？ | ⑤ 上下游系统会受什么影响？⑥ 现有日志/监控能定位到哪一步？⑦ 历史上有没有类似问题？ |
   | 项目结构优化 | ① 当前结构最痛的点是什么？② 目标架构的愿景是什么样？③ 能否分阶段迁移？④ 兼容性约束有哪些（老代码、老接口）？ | ⑤ 会影响团队协作方式吗？⑥ 构建/部署流程需要变吗？⑦ 是否需要保留旧路径过渡？ |
   | 中间件/工具创建 | ① 要解决的核心问题是什么？② 目标使用者是谁、使用场景有哪些？③ API 风格偏好（Builder/链式/函数式）？④ 有参考实现吗？ | ⑤ 扩展点怎么预留？⑥ 错误处理和日志约定是什么？⑦ 需要配套文档/示例吗？ |

   **输出格式**：
   ```
   🔍 需求深挖：

   基于 [场景名] 场景，我有以下问题需要确认：

   1. [问题1]
   2. [问题2]
   3. [问题3]

   请逐一回答，或直接告诉我"跳过"某个问题。
   ```

   ⛔ **门控**：等待用户回答后，才能继续。如果回答中又引出新问题，可追加一轮追问。

5. **需求完备性检查**：

   在进入步骤 2（代码探索）前，对已收集的信息做快速自检：

   **通用维度**（所有场景必查）：
   - □ 核心目标：一句话能说清楚要做什么
   - □ 成功标准：怎样算"做完了"
   - □ 改动边界：改什么、不改什么
   - □ 验证方式：怎么验证做对了

   **场景专属维度**（按当前场景勾选）：
   - [新功能] □ 数据流向 □ 用户交互流程 □ 异常场景
   - [优化] □ 性能基线 □ 目标指标 □ 可接受副作用
   - [修复] □ 复现路径 □ 紧急度 □ 回归范围
   - [重构] □ 目标结构 □ 迁移约束 □ 兼容性要求
   - [中间件] □ API 设计 □ 使用场景 □ 扩展性需求

   如果有维度未覆盖且对方案有影响，回到 1.4 补充追问。
   全部覆盖后，进入步骤 2。

📄 **文档输出**：将场景识别、需求理解、深挖结论写入计划文件的「背景」和「目标」章节。
→ 用户可查看文档并提出修改，修改完成后继续步骤 2。

### 步骤 2: 代码探索与现状分析

根据识别的场景，有针对性地探索代码库。

**通用探索动作**：
- 项目结构概览（关键目录和文件）
- 相关模块的代码阅读
- 依赖关系梳理
- 现有模式和约定识别

**按场景的专项探索**：

| 场景 | 专项探索 |
|------|---------|
| 新功能开发 | 找到扩展点、相似功能的实现方式、可复用的基础设施 |
| 性能优化 | 定位热点路径、分析调用链、识别潜在瓶颈 |
| 线上问题 | 追踪问题代码路径、分析异常处理链、检查边界条件 |
| 结构优化 | 绘制当前模块依赖图、识别循环依赖和不合理耦合 |
| 中间件创建 | 调研现有工具类、分析潜在使用场景、确定 API 风格 |

**探索产出**：

```
🔍 代码探索发现：
- 项目技术栈：[语言/框架/构建工具]
- 相关模块：[列出关键文件和模块]
- 现有模式：[项目中已采用的设计模式/约定]
- 可复用资源：[现有工具类、基础设施、抽象层]
- 关键代码路径：[核心调用链和数据流]
- 现有抽象层：[项目中可复用的工具类/基类/接口]
- 潜在风险：[发现的可能影响方案的因素]
```

向用户展示探索发现的关键信息，确保理解一致。

⛔ **门控**：以下情况需暂停，回到需求层面与用户确认：
- 发现与用户描述不一致的情况（如：用户说"新增功能"但实际需要改现有逻辑）
- 发现用户未提及但会受影响的模块（如：改 A 必须同时改 B）
- 发现需求中隐含的技术约束（如：现有数据结构不支持用户期望的功能）
- 发现多种实现路径，需要用户提供业务优先级来决策

📄 **文档输出**：将代码探索发现写入计划文件的「现状分析」章节。
→ 用户可查看文档确认探索发现是否准确，提出补充或修正。

⛔ **门控**：等待用户确认代码探索发现无误后，才能继续

### 步骤 3: 协作式方案构建

基于代码探索的结果，与用户协作确定技术方案。

1. **方案概述**：

   ```
   📐 方案概述：
   - 整体思路：[一句话描述方案核心]
   - 改动范围：[涉及哪些模块/文件]
   - 不改什么：[明确排除项]
   ```

2. **技术决策点**：

   识别方案中需要确定的技术决策，逐一与用户讨论。

   **处理规则**：
   | 情况 | 处理方式 |
   |------|----------|
   | 代码库中已有明确约定 | 遵循现有约定，告知用户即可 |
   | 用户指定了方案 | 直接采用，无需讨论 |
   | 有多个可选方案 | 列出选项 + ⭐推荐 + 理由，等待确认 |

   **输出格式**（需要用户决策时）：
   ```
   🔧 技术决策点：

   1. [决策主题]
      - A) [方案A描述]
      - B) [方案B描述]
      ⭐ 推荐 [X]：[推荐理由，结合代码探索发现]

   2. [决策主题]
      ...

   请确认或选择其他方案。
   ```

   ⛔ **门控**: 需用户确认技术决策后，才能继续

3. **任务拆解预览**：

   将方案拆解为可执行的任务列表，供用户审核。每个任务的描述要达到"不懂业务的技术人拿到就能写代码"的程度：

   ```
   📝 任务拆解：

   任务 1: [任务标题] [complexity: small]
   - 做什么：[具体描述，包含业务背景]
   - 为什么：[这个任务解决什么问题]
   - 改哪里：[精确到文件路径和方法名]
   - 怎么改：[实现思路，关键步骤，伪代码级别]
   - 参考实现：[项目中可参考的类似代码路径]
   - 验收标准：[可执行的验证步骤]
   - 依赖：无

   任务 2: [任务标题] [complexity: medium]
   - 依赖：任务 1
   ...

   任务间依赖关系：1 → 2 → 3
   预估总任务数：[N] 个
   ```

   ⛔ **门控**: 用户确认任务拆解合理后，才能继续

📄 **文档输出**：将方案概述、技术决策、任务拆解写入计划文件的「技术方案」和「任务列表」章节。
→ 用户可查看完整文档，对任何部分提出修改建议。

### 步骤 4: 完善方案文档

基于前面步骤已渐进写入计划文件的内容，进行最终完善（此时仍在 plan mode 中）。

**完善动作**：
1. 补充「风险与注意事项」章节
2. 检查文档完整性，确保各章节内容连贯
3. 确保「现状分析」章节包含关键代码路径和现有抽象层
4. 确保「技术方案」章节包含数据流图和接口设计
5. 确保每个任务的 JSON 中包含完整的 `implementationGuide`
6. 最终校对，确保文档可作为独立技术方案交付

📄 **文档输出**：将风险章节和完善后的内容更新到计划文件。
→ 用户可查看最终文档，确认所有章节完整、连贯。

**task.md 格式**：

```markdown
# [项目/功能名称] 技术方案

## 背景

[需求背景描述]

## 目标

[核心目标，1-3 句话]

## 现状分析

[基于代码探索的现状描述]
- 技术栈：[语言/框架]
- 相关模块：[关键文件清单]
- 现有模式：[项目约定]

### 关键代码路径

[列出核心调用链和数据流，帮助开发者快速理解代码运行逻辑]

### 现有抽象层

[项目中可复用的工具类/基类/接口，避免重复造轮子]

### 接口设计

[新增/修改的接口签名和参数说明]

### 技术决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| [决策1] | [选择] | [理由] |

### 改动范围

- 涉及：[模块/文件列表]
- 不涉及：[明确排除项]
```

### 步骤 5: 提交方案

确认计划文件内容完整（步骤 1-4 已渐进写入所有章节），调用 `ExitPlanMode` 提交方案供用户审批。

⛔ **门控: 等待用户审批方案**

### 步骤 6: 写入文件（用户审批后执行）

用户审批方案后，执行文件写入：

1. **写入 task.md 到项目根目录**

   将步骤 4 生成的完整内容写入 `task.md`。

2. **输出完成信息**：

   ```
   ✅ 技术方案已生成！

   已创建:
   • task.md - 技术方案文档（含 [N] 个任务定义）

   方案概要:
   → 场景：[场景类型]
   → 改动范围：[涉及模块]
   → 任务数：[N] 个
   → 技术决策：[已确认的关键决策]

   下一步:
   • 运行 /plan-init 基于 task.md 初始化项目
   • 或手动调整 task.md 后再初始化
   ```

3. ⛔ **写入完成后停止**

   ❌ **禁止**: 自动调用 `/plan-init`
   ❌ **禁止**: 自动创建 `features.json`
   ❌ **禁止**: 自动开始执行任务

## 特殊情况处理

### 用户提供需求文档

当用户提供 MD 文件路径、飞书文档链接或语雀文档链接时：

1. 读取文档内容
2. 提取需求要点
3. 按正常流程进入理解确认环节

### 用户要求写入飞书/语雀

如果用户明确要求将方案写入飞书或语雀（而非本地 task.md），在步骤 6 中：

- 飞书：使用飞书 MCP 工具创建文档并写入
- 语雀：使用语雀 MCP 工具创建文档并写入
- 同时仍创建本地 task.md 作为备份

### 需求变更

在步骤 1-4 的任何阶段，如果用户提出需求变更：

1. 确认变更内容
2. 评估对已完成分析的影响
3. 从受影响的步骤重新开始（不需要从头来过）
4. 更新计划文件：删除或修改受影响章节的内容，从受影响步骤的产出开始重写

### 快速模式（部分流程启动）

如果用户已完成某些前置工作，可跳过对应步骤：

| 用户已有 | 跳过 | 从哪开始 |
|---------|------|---------|
| 需求文档 + 代码分析报告 | 步骤 1-2 | 步骤 3（方案构建） |
| 完整方案草稿 | 步骤 1-3 | 步骤 4（完善文档） |
| 只需任务拆解 | 步骤 1-2 的部分内容 | 步骤 3 的任务拆解环节 |

跳过步骤时，仍需将用户已有内容写入计划文件对应章节，确保文档完整性。

## 与 /plan-init 的衔接

task.md 的任务列表部分采用与 `features.json` 兼容的 JSON 格式，用户可以：

1. 直接运行 `/plan-init`，将 task.md 作为需求文档输入
2. 手动调整 task.md 中的任务后再初始化
3. 选择性地将部分任务导入 `/plan-init`

⚠️ **关键原则**: `/plan-preview` 只负责方案输出，不触碰 `/plan-init` 的职责范围。

### category 兼容性说明

plan-preview 的 `category` 为 plan-init 的超集，兼容映射如下：

| plan-preview category | plan-init 是否支持 | 说明 |
|----------------------|-------------------|------|
| core | 支持 | 核心功能 |
| ui | 支持 | 界面相关 |
| feature | 支持 | 新功能 |
| optimization | 支持 | 性能/逻辑优化 |
| bugfix | 原样保留 | plan-init 会保留未识别的 category 值 |
| refactor | 原样保留 | plan-init 会保留未识别的 category 值 |
| middleware | 原样保留 | plan-init 会保留未识别的 category 值 |

### implementationGuide 字段说明

`implementationGuide` 是 plan-preview 的增强字段，为执行者提供精确的实现指引。
- plan-init 会原样保留此字段到 features.json（JSON 向前兼容）
- plan-next 执行时可参考此字段中的 targetFiles、approach 等信息加速理解
- 如果用户手动删除此字段，不影响 plan-init / plan-next 的核心流程
