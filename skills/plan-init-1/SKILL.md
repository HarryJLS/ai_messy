# Plan Init 1（一站式）

在 Plan Mode 中完成需求分析、技术决策和任务分解，审批后自动串联 `/plan-write` + `/plan-next` 完成全流程。

**本 skill 是 `/plan-init` 的一站式版本。** 审批后自动调用 `/plan-write` 写入文件，再自动调用 `/plan-next` 循环执行任务。

## 设计原则

- **抗遗忘**：通过读取相关任务日志恢复上下文
- **抗范围蔓延**：JSON 定义范围，日志提供详细上下文
- **可演进**：AI 可在执行前优化方案
- **精准执行**：只改该改的，不碰不该碰的
- **资深开发视角**：考虑复用性、扩展性、健壮性

## 核心文件

| 文件 | 用途 |
|------|------|
| `features.json` | 任务的单一事实来源（由 /plan-write 创建） |
| `dev-YYYY-MM-DD.log` | 统一开发日志（由 /plan-write 创建） |

## 协议

### 步骤 0: 进入 Plan Mode

skill 触发后，**立即调用 `EnterPlanMode` 工具**进入计划模式。

在 plan mode 中完成步骤 1-3 的所有探索、分析和任务分解工作。

### 步骤 1: 检查现有文件

检查项目根目录的 `features.json` 和 `dev-*.log` 文件。

**如果 `features.json` 存在**，询问用户：

| 选项 | 行为 |
|------|------|
| 覆盖 | 备份到 `features.backup.{timestamp}.json`，创建新文件 |
| 追加 | 向现有数组添加新任务 |
| 合并 | 保留现有，仅添加非重复项 |
| 取消 | 中止初始化 |

**如果 `dev-*.log` 文件存在**，告知用户："发现现有日志，将保留它们。"

⛔ **门控**: 在用户选择选项前不得继续

### 步骤 2: 规划文件创建

确认需要由 `/plan-write` 创建的文件清单：

1. `features.json` — 完整任务列表
2. `dev-YYYY-MM-DD.log` — 初始化日志头部 + 框架设置日志条目（YYYY-MM-DD 为当天日期）

### 步骤 3: 获取项目目标

**快速路径检测**：读取项目根目录 `task.md`，检查是否包含 JSON 任务列表。

| task.md 状态 | 处理方式 |
|-------------|----------|
| 不存在 | 正常流程（询问目标 → 澄清 → 决策 → 分解） |
| 存在但无 JSON 任务列表 | 作为需求文档输入，跳过"询问目标"，进入理解确认 |
| 存在且包含完整 JSON 任务列表 | **快速路径**：跳过需求收集和任务分解，直接提取 JSON → 写入计划文件 → ExitPlanMode |

快速路径时，仍需向用户展示提取到的任务摘要并确认。

---

1. **询问用户**: "这个项目的核心目标是什么？"
2. ⛔ **等待用户响应**

3. **理解确认**（仅当从文档读取需求时）：

**触发条件**：用户提供了需求文档路径或让你读取 MD 文件

```
📋 我的理解：
- 要做什么：[复述核心目标]
- 要改什么：[具体改动点，精确到类/方法]
- 不改什么：[明确排除项]
- 边界条件：[特殊情况处理]

❓ 不确定的地方：[列出理解模糊的点，主动追问]

请确认是否正确，或指出理解偏差。
```

⛔ **门控**：用户确认理解正确后，才能继续

4. **需求澄清**（资深开发视角）：

| 类别 | 追问示例 |
|------|----------|
| 复用性 | 有没有现成的原子能力/工具类可以复用？ |
| 影响范围 | 这个改动会影响哪些模块？有没有上下游依赖？ |
| 边界 | 只改 A 还是 A 和 B 都改？ |
| 兼容性 | 需要向后兼容吗？老接口怎么处理？ |
| 异常处理 | 失败场景怎么处理？ |
| 性能考量 | 性能要求？是否需要缓存？ |
| 测试策略 | 怎么验证改对了？怎么验证没改错？ |

⛔ **门控**：关键问题未澄清前，不得开始任务分解

5. **技术实现决策点识别**：

在分解任务前，识别实现时**必然要确定**的技术问题。

**处理规则**：
| 情况 | 处理方式 |
|------|----------|
| 用户指定了参考方法/工具 | 直接按参考方法的方式处理，无需询问 |
| 用户未指定 | 列出选项 + ⭐推荐 + 理由，等待确认 |

**常见决策点**：

| 场景 | 必问问题 |
|------|----------|
| 文件处理 | 先下载到本地还是流式处理？大文件怎么分片？ |
| 外部调用 | 同步还是异步？超时多久？失败重试几次？ |
| 数据转换 | 用哪个库解析？格式不符怎么处理？ |
| 并发操作 | 需要加锁吗？用什么锁？ |
| 状态存储 | 存内存、Redis、还是数据库？ |
| 配置项 | 硬编码还是可配置？配置放哪里？ |

**识别方法**：对每个任务问自己"写到这行代码时，我必须知道什么才能继续？"

**输出格式**（用户未指定参考方法时）：
```
🔧 技术决策点：

1. OSS 文件处理方式
   - A) 先下载到本地临时目录，处理完删除
   - B) 流式读取，边下边处理
   ⭐ 推荐 A：文件需要多次读取，流式会重复请求 OSS

2. 解析库选择
   - A) Jackson
   - B) Gson
   ⭐ 推荐 A：项目已有依赖，类型安全性更好

请确认或选择其他方案。
```

⛔ **门控**: 未指定参考方法时，需用户确认技术决策后，才能继续任务分解

6. **提取参考资源和数据样例**：

在分解任务前，从用户提供的需求文档/描述中识别并提取：

| 类型 | 说明 | 处理方式 |
|------|------|----------|
| 参考方法/工具 | 用户提到的现有方法、工具类、API、第三方库 | **必须**记录到任务的 `references` 字段，执行时需查找其源码/文档并按其封装方式使用 |
| 数据样例 | 用户提供的输入/输出数据、JSON 结构、报文格式等 | **必须**记录到任务的 `dataSamples` 字段，执行时需基于样例编写解析逻辑和验证用例 |
| 文档链接 | 用户提供的参考文档、Wiki、设计文档路径 | 记录到 `references` 字段，执行时需阅读并遵循 |

⚠️ **关键原则**：用户提供的参考方法和数据样例是**硬约束**，不可忽略或自行替代。

7. 将用户响应分解为原子任务:

```json
{
  "id": "1",
  "category": "core|ui|feature|optimization",
  "description": "清晰的任务描述（做什么 + 为什么）",
  "steps": ["具体步骤1", "具体步骤2"],
  "acceptance": ["验收标准1：可验证的结果", "验收标准2"],
  "boundary": "可选：明确边界（只改什么，不改什么）",
  "test": "unit|integration|e2e|manual: 简述测试方法和关键用例",
  "references": ["可选：用户指定的参考方法/工具/API/文档，如 XxxUtil.parseData()、某某文档路径"],
  "dataSamples": ["可选：用户提供的数据样例描述或路径，如 '见需求文档中的 JSON 报文示例'"],
  "passes": false
}
```

8. **将任务列表写入计划文件并退出 Plan Mode**
   - 如有 `references` 或 `dataSamples`，需在计划中标注
   - 将完整的任务列表、步骤 1 的文件检查结果、步骤 2 的文件创建计划，一并写入计划文件
   - **计划文件中的任务列表必须是完整的 JSON 格式**，供 `/plan-write` 直接提取写入
   - 调用 `ExitPlanMode` 提交计划供用户审批

9. ⛔ **门控: 等待用户审批计划**

### 步骤 4: 自动串联 plan-write + plan-next

用户审批计划后：

1. 输出 "✅ 计划已审批！正在写入文件..."
2. **自动调用 `Skill('plan-write')` 完成文件写入**（features.json + 开发日志）
3. 写入完成后，**自动调用 `Skill('plan-next')` 开始循环执行任务**

## 完成后输出

见步骤 4。

## 后续修改同步

如果用户在初始化后要求修改任务（增删改），必须同步更新 `features.json`：

| 操作 | 处理方式 |
|------|----------|
| 新增任务 | 添加到 `features.json` 数组末尾，分配新 ID |
| 修改任务 | 更新 `features.json` 中对应任务的字段 |
| 删除任务 | 从 `features.json` 中移除对应任务 |
| 重排顺序 | 更新 `features.json` 中任务的顺序和 ID |

⚠️ **关键原则**: `features.json` 是任务的单一事实来源，任何修改必须立即同步到文件。
