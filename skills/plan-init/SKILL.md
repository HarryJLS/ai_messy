# Plan Init

在 Plan Mode 中完成需求分析、技术决策和任务分解，输出计划文件供审批。

**本 skill 只负责"想"。** 审批后输出确认信息并停止，串联逻辑由调用方决定。

## 三档自适应

根据输入清晰度自动选择工作深度，无需用户手动选模式：

| 输入 | 模式 | 行为 |
|------|------|------|
| 模糊需求（口头描述） | 深度模式 | 需求深挖 → 场景识别 → 代码探索 → 方案构建 → 任务分解 |
| 明确技术文档（task.md 无 JSON） | 标准模式 | 理解确认 → 技术决策 → 任务分解 |
| 已有 JSON 任务列表（task.md 含 JSON） | 极速模式 | 提取 JSON → 确认 → 完成 |

**判断逻辑**：

1. 读取 `.plan/task.md`
2. 如果存在且含完整 JSON 任务列表 → **极速模式**
3. 如果存在但无 JSON → **标准模式**（以文档为需求输入）
4. 如果不存在 → 询问用户目标：
   - 用户提供了详尽的技术描述（含改动范围、实现思路） → **标准模式**
   - 用户只给了笼统目标或一句话需求 → **深度模式**

告知用户当前采用的模式及原因，用户可要求切换。

## 设计原则

- **抗遗忘**：通过读取相关任务日志恢复上下文
- **抗范围蔓延**：JSON 定义范围，日志提供详细上下文
- **可演进**：AI 可在执行前优化方案
- **精准执行**：只改该改的，不碰不该碰的
- **资深开发视角**：考虑复用性、扩展性、健壮性
- **架构师视角**（深度模式）：深入代码库理解现状后给出专业方案
- **代码驱动**（深度模式）：方案基于实际代码探索，而非凭空设计

## 核心文件

| 文件 | 用途 |
|------|------|
| `.plan/task.md` | 技术方案文档（深度模式产出，或用户提供） |
| `.plan/features.json` | 任务的单一事实来源（由 /plan-write 创建） |
| `.plan/dev-YYYY-MM-DD.log` | 统一开发日志（由 /plan-write 创建） |

## 五大场景（深度模式）

| 场景 | 核心关注点 | 探索重点 |
|------|-----------|---------|
| 新功能开发 | 在哪加？怎么加？影响什么？ | 相关模块结构、现有抽象、扩展点 |
| 性能/逻辑优化 | 瓶颈在哪？怎么改？改了会不会引入新问题？ | 热点代码路径、调用链、数据流 |
| 线上问题修复 | 根因是什么？修复方案？如何防止复发？ | 问题代码、上下游依赖、异常处理链 |
| 项目结构优化 | 当前结构问题？目标结构？迁移路径？ | 模块划分、依赖关系、分层架构 |
| 中间件/工具创建 | 解决什么问题？边界在哪？API 怎么设计？ | 现有工具类、使用场景、接口契约 |

## 协议

### 步骤 0: 进入 Plan Mode

skill 触发后，**立即调用 `EnterPlanMode` 工具**进入计划模式。

在 plan mode 中完成所有探索、分析和任务分解工作。

**渐进式文档输出**（深度模式）：
- 每个步骤完成后，将该步骤的结构化产出写入计划文件（plan file）
- 计划文件使用 .plan/task.md 的目标格式组织内容
- 用户可随时查看文档并提修改建议

### 步骤 1: 检查现有文件

检查 `.plan/` 目录下的 `features.json` 和 `dev-*.log` 文件。

**如果 `.plan/features.json` 存在**，询问用户：

| 选项 | 行为 |
|------|------|
| 覆盖 | 备份到 `.plan/features.backup.{timestamp}.json`，创建新文件 |
| 追加 | 向现有数组添加新任务 |
| 合并 | 保留现有，仅添加非重复项 |
| 取消 | 中止初始化 |

**如果 `.plan/dev-*.log` 文件存在**，告知用户："发现现有日志，将保留它们。"

⛔ **门控**: 在用户选择选项前不得继续

### 步骤 2: 获取需求 + 判断模式

**读取 `.plan/task.md`**（如存在），检查是否包含 JSON 任务列表。

| task.md 状态 | 模式 | 下一步 |
|-------------|------|--------|
| 存在且含完整 JSON 任务列表 | **极速模式** | 提取 JSON → 向用户展示任务摘要 → 跳到步骤 7 |
| 存在但无 JSON | **标准模式** | 作为需求文档输入 → 理解确认 → 跳到步骤 5 |
| 不存在 | 询问用户目标 | 根据输入丰富度判断深度/标准模式 |

**询问用户**（仅 task.md 不存在时）：

```
请描述你要做的事情：
- 可以直接说需求
- 可以提供需求文档路径（我会读取并分析）
- 可以提供飞书/语雀文档链接
```

⛔ **门控**: 等待用户提供需求信息

**模式判断**（用户提供需求后）：

- 用户输入含具体改动范围、实现思路、文件路径等 → **标准模式** → 跳到步骤 5
- 用户输入只有笼统目标、一句话描述 → **深度模式** → 继续步骤 3

告知用户：`🎯 模式：[深度模式/标准模式]，原因：[简述]`
用户可要求切换模式。

### 步骤 3: 需求深挖 + 代码探索（仅深度模式）

#### 3.1 场景识别

根据需求自动识别所属场景：

```
🎯 场景识别：[新功能开发 / 性能优化 / 线上问题修复 / 项目结构优化 / 中间件创建]
理由：[简述]
```

如果场景判断不确定，列出候选场景让用户选择。

#### 3.2 理解确认

```
📋 我的理解：
- 要做什么：[复述核心目标]
- 核心诉求：[用户最关心的点]
- 预期产出：[最终交付物]

❓ 不确定的地方：[列出理解模糊的点，主动追问]

请确认是否正确，或指出理解偏差。
```

⛔ **门控**：用户确认理解正确后，才能继续

#### 3.3 需求深挖

根据识别的场景，从问题库中挑选关键问题与用户确认细节，详见 [references/scenario-questions.md](references/scenario-questions.md)。

**提问原则**：
- 只问对方案有实质影响的问题
- 跳过用户在需求描述中已明确回答的问题
- 每轮最多 3-5 个问题，避免问答疲劳

⛔ **门控**：等待用户回答

#### 3.4 需求完备性检查

**通用维度**（所有场景必查）：
- □ 核心目标：一句话能说清楚要做什么
- □ 成功标准：怎样算"做完了"
- □ 改动边界：改什么、不改什么
- □ 验证方式：怎么验证做对了

**场景专属维度**：
- [新功能] □ 数据流向 □ 用户交互流程 □ 异常场景
- [优化] □ 性能基线 □ 目标指标 □ 可接受副作用
- [修复] □ 复现路径 □ 紧急度 □ 回归范围
- [重构] □ 目标结构 □ 迁移约束 □ 兼容性要求
- [中间件] □ API 设计 □ 使用场景 □ 扩展性需求

如果有维度未覆盖且对方案有影响，回到 3.3 补充追问。

📄 **文档输出**：将场景识别、需求理解、深挖结论写入计划文件的「背景」和「目标」章节。

#### 3.5 代码探索

根据场景有针对性地探索代码库。

**通用探索动作**：
- 项目结构概览（关键目录和文件）
- 相关模块的代码阅读
- 依赖关系梳理
- 现有模式和约定识别

**按场景的专项探索**：

| 场景 | 专项探索 |
|------|---------|
| 新功能开发 | 找到扩展点、相似功能的实现方式、可复用的基础设施 |
| 性能优化 | 定位热点路径、分析调用链、识别潜在瓶颈 |
| 线上问题 | 追踪问题代码路径、分析异常处理链、检查边界条件 |
| 结构优化 | 绘制当前模块依赖图、识别循环依赖和不合理耦合 |
| 中间件创建 | 调研现有工具类、分析潜在使用场景、确定 API 风格 |

**探索产出**：
```
🔍 代码探索发现：
- 项目技术栈：[语言/框架/构建工具]
- 相关模块：[列出关键文件和模块]
- 现有模式：[项目中已采用的设计模式/约定]
- 可复用资源：[现有工具类、基础设施、抽象层]
- 关键代码路径：[核心调用链和数据流]
- 潜在风险：[发现的可能影响方案的因素]
```

⛔ **门控**：以下情况需暂停，回到需求层面与用户确认：
- 发现与用户描述不一致的情况
- 发现用户未提及但会受影响的模块
- 发现需求中隐含的技术约束
- 发现多种实现路径，需要用户提供业务优先级来决策

📄 **文档输出**：将代码探索发现写入计划文件的「现状分析」章节。

⛔ **门控**：等待用户确认代码探索发现无误后，才能继续

### 步骤 4: 协作式方案构建（仅深度模式）

基于代码探索的结果，与用户协作确定技术方案。

#### 4.1 方案概述

```
📐 方案概述：
- 整体思路：[一句话描述方案核心]
- 改动范围：[涉及哪些模块/文件]
- 不改什么：[明确排除项]
```

#### 4.2 技术决策点

识别方案中需要确定的技术决策，逐一与用户讨论。

**处理规则**：
| 情况 | 处理方式 |
|------|----------|
| 代码库中已有明确约定 | 遵循现有约定，告知用户即可 |
| 用户指定了方案 | 直接采用，无需讨论 |
| 有多个可选方案 | 列出选项 + ⭐推荐 + 理由，等待确认 |

⛔ **门控**: 需用户确认技术决策后，才能继续

#### 4.3 任务拆解预览

将方案拆解为可执行的任务列表，供用户审核：

```
📝 任务拆解：

任务 1: [任务标题] [domain: backend] [complexity: small]
- 做什么：[具体描述，包含业务背景]
- 为什么：[这个任务解决什么问题]
- 改哪里：[精确到文件路径和方法名]
- 怎么改：[实现思路，关键步骤]
- 参考实现：[项目中可参考的类似代码路径]
- 验收标准：[可执行的验证步骤]
- API 契约：[全栈项目后端任务需列出前端会用到的接口]
- 依赖：无
```

**依赖设置原则**：只有**编译级依赖**才设 dependsOn（如任务 B import 了任务 A 新增的类）。前后端不同 domain 的任务、服务间 HTTP/RPC 调用等运行时依赖不设 dependsOn，用 apiContracts 约定契约即可。详见步骤 5.5 的「dependsOn 智能分析规则」。

⛔ **门控**: 用户确认任务拆解合理后，才能继续

📄 **文档输出**：将方案概述、技术决策、任务拆解写入计划文件的「技术方案」和「任务列表」章节。

→ 继续到步骤 6

### 步骤 5: 需求澄清 + 技术决策 + 任务分解（标准模式，或深度模式跳过步骤 3-4 时）

#### 5.1 理解确认

```
📋 我的理解：
- 要做什么：[复述核心目标]
- 要改什么：[具体改动点，精确到类/方法]
- 不改什么：[明确排除项]
- 边界条件：[特殊情况处理]

❓ 不确定的地方：[列出理解模糊的点，主动追问]

请确认是否正确，或指出理解偏差。
```

⛔ **门控**：用户确认理解正确后，才能继续

#### 5.2 需求澄清（资深开发视角）

| 类别 | 追问示例 |
|------|----------|
| 复用性 | 有没有现成的原子能力/工具类可以复用？ |
| 影响范围 | 这个改动会影响哪些模块？有没有上下游依赖？ |
| 边界 | 只改 A 还是 A 和 B 都改？ |
| 兼容性 | 需要向后兼容吗？老接口怎么处理？ |
| 异常处理 | 失败场景怎么处理？ |
| 性能考量 | 性能要求？是否需要缓存？ |
| 测试策略 | 怎么验证改对了？怎么验证没改错？ |

⛔ **门控**：关键问题未澄清前，不得开始任务分解

#### 5.3 技术实现决策点识别

在分解任务前，识别实现时**必然要确定**的技术问题。

**处理规则**：
| 情况 | 处理方式 |
|------|----------|
| 用户指定了参考方法/工具 | 直接按参考方法的方式处理，无需询问 |
| 用户未指定 | 列出选项 + ⭐推荐 + 理由，等待确认 |

**识别方法**：对每个任务问自己"写到这行代码时，我必须知道什么才能继续？"

⛔ **门控**: 未指定参考方法时，需用户确认技术决策后，才能继续任务分解

#### 5.4 提取参考资源和数据样例

从用户需求中识别参考方法/工具（→ `references` 字段）、数据样例（→ `dataSamples` 字段）、文档链接（→ `references` 字段）。用户提供的参考方法和数据样例是**硬约束**，不可忽略或自行替代。

#### 5.5 任务分解

将用户响应分解为原子任务:

```json
{
  "id": "1",
  "domain": "backend|frontend",
  "app": "order-service",
  "appPath": "../order-service",
  "dependsOn": [],                      // 同应用引用：纯 ID，如 ["1", "2"]；跨应用引用：app:id 格式，如 ["order-api:3"]
  "complexity": "small|medium|large|trivial",
  "category": "core|ui|feature|optimization|bugfix|refactor|middleware",
  "description": "清晰的任务描述（做什么 + 为什么）",
  "steps": ["具体步骤1", "具体步骤2"],
  "implementationGuide": {
    "targetFiles": ["src/service/XxxService.java:processData()"],
    "approach": "实现思路描述",
    "referenceCode": ["src/service/YyyService.java:similarMethod()"],
    "dataFlow": "数据从哪来 → 经过什么处理 → 到哪去",
    "keyInterfaces": ["需要实现/调用的关键接口说明"]
  },
  "apiContracts": [],                      // ⭐ API 契约字段（涉及接口对接时必须生成）
  "acceptance": ["验收标准1：可验证的结果"],
  "boundary": "可选：明确边界（只改什么，不改什么）",
  "test": "unit|integration|e2e|manual: 简述测试方法和关键用例",
  "references": ["可选：用户指定的参考方法/工具/API/文档"],
  "dataSamples": ["可选：用户提供的数据样例描述或路径"],
  "passes": false
}
```

**domain 字段**：
- 纯后端/纯前端项目可省略（默认根据项目类型推断）
- 全栈项目必须标注每个任务的 domain

**app 字段**：
- 单应用项目可省略
- 多应用/微服务项目必须标注每个任务所属的应用名（如 `order-service`、`user-service`、`admin-web`）
- 配合 `appPath` 字段指定应用的项目路径（相对路径或绝对路径）
- 跨会话执行时，backend-single/frontend-single 支持按 app 过滤，自动路由到对应项目目录执行

**dependsOn 字段**：
- 同应用内依赖：使用纯任务 ID，如 `["1", "2"]`
- 跨应用依赖：使用 `app:id` 格式，如 `["order-api:3"]`，其中 `app` 必须与被引用任务的 `app` 字段一致
- 混合使用：`["1", "order-api:3"]`
- plan-next 执行时会检查 dependsOn 的任务是否已完成，未完成则阻塞并提醒用户先执行依赖任务

**dependsOn 智能分析规则**：

dependsOn 是硬阻塞——设错会导致不必要的串行等待，漏设会导致编译失败。分解任务时，按以下规则判断是否需要设置依赖：

| 场景 | 是否需要 dependsOn | 理由 |
|------|-------------------|------|
| 前端任务 vs 后端任务（不同 domain） | ❌ 不需要 | 前后端可并行开发，前端通过 apiContracts 约定接口，不依赖后端代码完成 |
| 同 domain 同 app 内，任务 B 使用任务 A 新增的类/方法 | ✅ 需要 | 编译级依赖，A 不完成 B 无法编译 |
| 同 domain 不同 app，服务 B 调用服务 A 的接口 | ❌ 不需要 | 服务间通过 HTTP/RPC 调用，运行时依赖而非编译依赖，可并行开发 |
| 业务 app 依赖 share/common 包的新增类/接口 | ✅ 需要 | share 包提供的类/接口是编译级依赖，必须先开发 share 包 |
| 同 app 内，任务 B 修改的文件依赖任务 A 修改的文件 | ✅ 需要 | 同一代码库内的直接代码依赖 |
| 两个任务改不同文件、无调用关系 | ❌ 不需要 | 无交集，可并行 |

**核心原则**：只有**编译级依赖**（import/引用了对方新增的类、方法、接口定义）才设 dependsOn。运行时依赖（HTTP 调用、消息队列、配置引用）不设 dependsOn，用 apiContracts 约定契约即可。

**多应用典型模式**：
```
share 包任务（定义公共接口/DTO）
  ↓ dependsOn（编译依赖）
业务服务 A 任务（实现接口 + 业务逻辑）    业务服务 B 任务（调用接口 + 业务逻辑）
                                          ↑ 无 dependsOn（HTTP 调用，运行时依赖）
前端任务（通过 apiContracts 对接）
  ↑ 无 dependsOn（不同 domain，可并行）
```

**apiContracts 字段**（涉及接口对接时必须生成）：

⚠️ 当任务涉及接口对接时必须生成 apiContracts：
- **全栈项目**：后端任务新增/修改接口且前端任务会调用 → 在后端任务中定义 apiContracts
- **纯前端项目**：前端任务调用后端 API → 在前端任务中记录要对接的接口规格（method、path、request、response）
- apiContracts 是接口对接的硬依赖——缺失会导致开发时接口格式不明确

当项目涉及接口对接时，需要在任务分解阶段就敲定接口契约，确保开发时有明确的接口规格可依赖。

```json
"apiContracts": [
  {
    "method": "GET|POST|PUT|DELETE",
    "path": "/api/users",
    "description": "获取用户列表",
    "request": {
      "query": { "page": "number", "size": "number" },
      "body": {}
    },
    "response": {
      "code": 200,
      "body": { "list": "User[]", "total": "number" }
    }
  }
]
```

只需定义会用到的接口，已有且不变的接口无需重复。全栈项目中，前端任务的 `implementationGuide.keyInterfaces` 应引用对应后端任务 ID 的 apiContracts。

深度模式产出的任务自带 `implementationGuide`（来自步骤 4 的代码探索）；标准模式中 `implementationGuide` 为可选，有就填。

### 步骤 6: 完善文档 + 写入计划文件 + ExitPlanMode

#### 深度模式

基于前面步骤已渐进写入计划文件的内容，进行最终完善：

1. 补充「风险与注意事项」章节
2. 检查文档完整性，确保各章节内容连贯
3. 确保「现状分析」章节包含关键代码路径和现有抽象层
4. 确保每个任务的 JSON 中包含完整的 `implementationGuide`
5. **应用覆盖性检查**（多应用/全栈项目必做）
6. 最终校对

#### 应用覆盖性检查（多应用/全栈项目）

当需求涉及多个应用或多个技术栈时，必须在最终校对前执行此检查，防止遗漏某个应用的任务：

1. **回溯需求来源**：重新审视用户需求文档/描述，提取需求中**明确提到的所有应用/项目路径**
2. **比对任务列表**：检查任务列表中的 `app`/`appPath` 是否覆盖了需求中提到的每个应用
3. **补漏**：如果某个应用在需求中被提及但任务列表中没有对应任务，必须补充

```
🔍 应用覆盖性检查：
需求涉及的应用：[列出需求中明确提到的所有应用]
任务列表覆盖的应用：[列出任务中所有 app 值]
未覆盖的应用：[列出差异] ← 需要补充任务
```

**常见遗漏场景**：
- 需求文档提到了前端页面修改，但只分解了后端任务
- 需求涉及 share 包（公共接口定义），但只关注了业务服务
- 需求提到多个微服务，但只分解了其中部分的任务

📄 完善后更新计划文件，确保格式符合 [references/task-template.md](references/task-template.md)。

#### 标准模式

将完整的任务列表、步骤 1 的文件检查结果、步骤 2 的文件创建计划，一并写入计划文件。

**计划文件中的任务列表必须是完整的 JSON 格式**，供 `/plan-write` 直接提取写入。

#### 所有模式

调用 `ExitPlanMode` 提交计划供用户审批。

⛔ **门控: 等待用户审批计划**

### 步骤 7: 审批后输出

用户审批计划后：

**深度模式**：写入 `.plan/task.md`（使用 `mkdir -p .plan` 确保目录存在）。

**所有模式**：输出确认信息并停止：

```
✅ 计划已审批！

已创建:
• .plan/task.md - 技术方案文档（含 [N] 个任务定义）  ← 仅深度模式

下一步:
• 运行 /plan-write 写入 .plan/features.json 和开发日志
• 运行 /plan-next 开始循环执行任务
```

⛔ 输出后立即停止，不得自动调用任何 skill。

❌ **禁止**: 自动调用 `/plan-write`
❌ **禁止**: 自动创建 `.plan/features.json`
❌ **禁止**: 自动开始执行任务

## 特殊情况处理

### 用户提供需求文档

当用户提供 MD 文件路径、飞书文档链接或语雀文档链接时：
1. 读取文档内容
2. 提取需求要点
3. 按正常流程进入理解确认环节

### 用户要求写入飞书/语雀

如果用户明确要求将方案写入飞书或语雀（而非本地 .plan/task.md），在步骤 7 中：
- 飞书：使用飞书 MCP 工具创建文档并写入
- 语雀：使用语雀 MCP 工具创建文档并写入
- 同时仍创建本地 .plan/task.md 作为备份

### 需求变更

在任何步骤中，如果用户提出需求变更：
1. 确认变更内容
2. 评估对已完成分析的影响
3. 从受影响的步骤重新开始
4. 更新计划文件中受影响的章节

### 后续修改同步

如果用户在初始化后要求修改任务（增删改），必须同步更新 `.plan/features.json`：

| 操作 | 处理方式 |
|------|----------|
| 新增任务 | 添加到 `.plan/features.json` 数组末尾，分配新 ID |
| 修改任务 | 更新 `.plan/features.json` 中对应任务的字段 |
| 删除任务 | 从 `.plan/features.json` 中移除对应任务 |
| 重排顺序 | 更新 `.plan/features.json` 中任务的顺序和 ID |

⚠️ **关键原则**: `.plan/features.json` 是任务的单一事实来源，任何修改必须立即同步到文件。

## 与 /plan-write 的衔接

.plan/task.md 的任务列表部分采用与 `.plan/features.json` 兼容的 JSON 格式。

category 和 implementationGuide 的兼容性说明详见 [references/compatibility.md](references/compatibility.md)。

## 完成后输出

见步骤 7 的输出格式。
